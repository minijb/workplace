## 一、如何评价一个 Skill —— PCSAID 评价框架

> 本节回答"我希望你给出一份如何评价一个 skill 的方法"。该框架既是本次评审的标尺，也是一份可复用的通用方法。

### 1.1 核心立论

一个 skill 的本质是**「被执行体（LLM / agent）按需加载、用于在特定场景规约自身行为的『知识 + 指令』包」**。因此：

> **评价一个 skill ≠ 评判文档写得是否美观，而 = 评判「加载该 skill 之后，执行体能否正确、可靠、高效地达成 skill 声明的目标，且不越界、不退化」。**

三条推论：
- 评价必须从「加载后的可观测行为」**倒推**，而非正向评文档；
- 每条结论须**有证据**（引用原文）、**可判定**（有明确通过/不通过判据）、**可操作**（给出能落地的改进）；
- 必须考虑**退化风险**——执行体的天然倾向（如"一次多做""替用户决策"）会在哪里压垮 skill 的约束。

### 1.2 六个评价维度（PCSAID）

| 维度 | 判定问句 | 典型反模式 |
|---|---|---|
| **P 定位与触发** | name/description 是否准确刻画场景与触发词？是否自声明与兄弟 skill 的边界（做什么/不做什么对仗）？有无职责重叠或真空？ | 边界模糊、靠语义猜触发、与同类 skill 职责交叉 |
| **C 内容与方法论** | 核心方法论是否有真源/依据、是否自洽？内部分类是否完备（MECE）、有无遗漏关键分支？流程是否闭环（前置→执行→收敛→完成标准）？ | 分类重叠或遗漏、流程缺收敛或验收、方法无依据却自封权威 |
| **S 结构与渐进式披露** | 入口是否精简自包含？长内容是否正确下沉？**何时加载下沉内容是否明示（不冗余、不缺失）**？引用基点是否统一、是否符合单一真源？ | 入口塞满细节、下沉内容本该在入口、同一内容多处重复、关键节从未被挂载 |
| **A 可执行性与遵从度** | 每条指令是否具体可判定？完成标准是否可被外部观测？硬约束是否有破口？边缘/失败路径是否有明文处置？退化风险有无锚点拉回？ | 含糊指令、不可观测的完成标准、对执行体天然倾向无对抗约束 |
| **I 一致性** | 入口与下沉内容有无矛盾？完成标准与正文流程是否对齐？术语是否全文统一？ | 同一规则两处表述不一、术语漂移、验收项在正文无对应 |
| **D 工程纪律** | 声明的硬约束（如只读/不落盘/一次一题）是否全程无破口？是否遵守工作区既定契约？ | 声称只读却隐含写入、约束前后不一 |

### 1.3 评分尺度与权重

- 每维 **0–5 分**：0 缺失 / 1 严重缺陷 / 2 明显不足 / 3 基本可用有瑕疵 / 4 良好 / 5 卓越。
- **权重**（反映"对加载后行为正确性的贡献"）：A、C 各 25%；P 20%；S、I、D 各 10%。
- 加权汇总 → 总分（0–100）→ 等级：≥85 优秀 / 70–84 良好 / 55–69 合格 / <55 待改进。

### 1.4 评价流程（可复现）

1. **读全**：frontmatter + 入口正文 + 全部 references + 其引用的 lib 契约（核实是否真单一真源）。
2. **提取契约**：skill 自声明的【目标】【完成标准】【不做什么】【边界】——这是验收基线。
3. **六维逐评**：每项给【证据（引用原文）】【评分】【改进】。
4. **交叉一致性**：入口 vs 下沉内容、完成标准 vs 正文流程、术语统一性。
5. **真源核对**：若 skill 声称改编/依据外部方法，**联网核对忠实度与增删得失**。
6. **退化推演**：站在执行体视角枚举"可能在哪里悄悄偏离"，检查锚点。
7. **产出**：六维评分表 + 优点 + 关键问题（按严重度）+ **优先级改进清单（P0/P1/P2）**。

### 1.5 评价自身的质量守则（反模式清单）

- 禁只褒不贬或只贬不褒；禁发现无证据（"感觉啰嗦"）；禁建议不可操作（"加强结构"）。
- 禁用代码项目标准评知识包（强求测试覆盖率等——不适用）。
- 禁把 skill 当孤立产品评，忽略其与兄弟 skill 的边界与网络位置。
- （本框架自身遵循它对 skill 的要求：每维有可判定问句、每发现需证据、流程可复现。）

### 1.6 适用范围

适用：LLM/Agent 的 skill / prompt / 指令包 / SOP 知识包。不适用：普通库代码、配置文件（用常规代码评审）。

## 二、五视角评价

### 视角一 · 内容与方法论严谨性（C 维） — 80 / 良好

**总评**：盘问法四支柱（一次一题 / 每题推荐答案 / 事实自查 vs 决策交用户 / 共识前不动工）有显式理由、失效模式覆盖充分、认识论纪律严格，方法论本身站得住。短板在六类分支被同时声明为「穷尽判据」却非严格 MECE，且漏掉「权限/授权」一等轴、偏数据建模视角却无按领域裁剪指引。

**优点**
- 方法论整体站得住：一次一题给了明确 why（后答依赖前答，多问让推荐答案失据）；「能查就别问」把可查事实与需用户决策清晰二分，是少见的认识论纪律；§4 显式覆盖死锁/疲劳/琐碎/跑题等失效模式；§3 要求纯主观标注「[无事实依据，仅经验判断]」防主观当客观。

**关键问题**
- **【中】六类非严格 MECE 却被当「穷尽判据」**：②关系模型（基数/归属）与④存放索引（唯一键/去重）在键/归属上实质重叠；③生命周期（创建/流转/销毁）与⑥边缘场景（并发/失败回滚）在状态迁移上重叠。把有重叠的分类当穷尽判据有逻辑张力——同一问题可能被两类各计一次"已有结论"，或被两类都漏掉。
- **【中】缺「权限/授权/安全」一等维度**：「谁能对 X 做什么操作、鉴权/授权边界、特权隔离」未被任一类覆盖，①②③ 都只从数据归属角度触及，替代不了访问控制策略。这不是边缘场景，是主轴。
- **【中】六类偏「数据建模」视角、缺按领域裁剪指引**：③只覆盖实例的创建/流转/销毁，不覆盖 schema/数据的演进、版本兼容与迁移；一致性/时序/可观测性只零星折叠进⑥。却未说明它适用于哪类设计、对其他类设计该如何增补类别。
- **【低】事实自查支柱偏代码中心**：宿主是「非代码知识工作区」，声明范围是「计划/决策/想法」，许多盘问对象无源码可查。「代码」应泛化为「既有上下文（代码/文档/既有结论/工具输出）」。
- **【低】推荐答案存在锚定风险**，skill 未显式承认这一权衡（橡皮图章式同意），也未给"高利害/用户强主见决策改用开放问法"的逃生阀。

### 视角二 · 结构与渐进式披露（S 维） — 70 / 良好

**总评**：入口骨架与 references 头部卫生属上乘（骨架与 plan-create/plan-execute 完全一致，references 头部用「适用范围 + 访问方式」显式标注何时加载与两种 URI）。但渐进式披露有两个真实结构性缺陷：references §3/§4 两节从未被 SKILL.md 挂载（孤儿按需内容）；SKILL.md §2 与 references §2 给出两套未对齐的施压分类。整体「骨架对、挂载漏、预览偏」。

**优点**
- 入口骨架遵循工作区分层约定；「不做什么」把与 4 个兄弟的边界划得清晰；references 头部是兄弟 skill 里较好的按需加载卫生实践；六类表正确下沉。

**关键问题**
- **【中】references §3/§4 从未被入口挂载，形成孤儿按需内容**：SKILL.md 全文仅在 §1 挂载 references §1、§2 挂载 references §2，共两处；references §3「推荐答案怎么给」与 §4「节奏与收口」无任何入口指针。一个只读 SKILL.md 的 LLM 不会知道去加载这两节——§4 的死锁/跑题逃逸从入口侧不可达。对照：兄弟 skill（plan-create/plan-execute）每个 reference 都显式「见 X.md——前先 read」挂载。
- **【中】SKILL.md §2 与 references §2 对「施压」给出两套未对齐的分类**：入口列 4 动作（对撞/收窄/拿场景/用代码），references §2 是 5 探针（反例/边界/冲突/代价/替换）。两套只有部分对应，「反例/代价/替换」在入口无对应，「收窄模糊用语/用代码印证」在 references §2 无对应。渐进式披露的契约是"入口摘要须忠实预览细节"，此处摘要与细节分叉。
- **【中】节奏主路径在入口缺配平的失败路径处置**：一次一题的主路径在入口侧没有死锁/跑题逃逸（只藏在未挂载的 §4），违反"每条主路径配平边缘/失败路径"的工作区纪律。

### 视角三 · 外部真源对齐（联网） — 80 / 良好(B+)

**总评**：对 mattpocock 原方法的【核心忠实度很高】——原 SKILL.md 全部五要素（无情逐题/设计树按依赖逐支/能查代码就不问/每题推荐答案/达成共享理解才止步）+ v1.1 两项硬修复（确认闸门、Facts vs Decisions）都被忠实保留并译得准确。真正价值在两处**实质增强**：①六类把原方法只说「walk down each branch」的模糊指令落到可操作穷尽判据——正面回应 HN 社区"等于 be critical"的核心批评；②五探针把「relentless」从口号落到手法，与 Socratic elenchus 反例归谬传统同源。三处可修复风险：源漂移归因、使用时机丢失、归因拔高。

**优点**
- **核心五要素 + v1.1 两项修复全部忠实保留**，归因准确无张冠李戴。
- **六类分支覆盖是对原方法"措辞空泛"缺点的实质增强**：HN 批评者认为「walk down the design tree 这类华丽措辞对 LLM 可能等于 be critical」；维护者反驳的关键恰是"它有层级与依赖的具体含义"。六类把这句抽象话落成了具体维度。
- **五探针把「relentless」从形容词落成动词清单**，且「力度拿捏：逼精确不是抬杠」的自约束避免了"无情"滑向咄咄逼人。
- **横向定位独到**：同时具备 rubber duck（对话性）、Socratic（反例施压）、DDD（术语定界）三者长处——rubber duck 是单向自述无追问、Socratic 无推荐答案无事实自查、DDD 只管术语不管决策树。

**关键问题**
- **【中】源漂移归因**：mattpocock 在 v1.1（2026-07-08）把盘问技术抽成单一真源 `grilling` 原语，`grill-me` 降为它的两个「前门」之一；且 Matt 公开声明「不再用 grill-me 做编码，改推 grill-with-docs / domain-model」。grilling-me 的归因仍停留在已被降级的旧 artifact，读者按其查源会产生认知落差。
- **【中】使用时机丢失**：原方法明确列出五类触发场景（写 PRD 前/让 agent 实现前/定数据模型或 API 形状前/多决策相互依赖/想要被反驳而非附和），grilling-me 删成笼统的「动工前」，丢了路由精度，description 触发词也偏弱。
- **【低】归因拔高**：把原作者自谦为「最近加的一行便利」的推荐答案，升格为「mattpocock 盘问法的硬纪律」并单列格式规范（方向被 v1.1 证实正确，但属诠释性拔高，应标注为本地补强）。
- **【低】通用性收窄**未声明：原方法自诩 universal tool（含非编码场景），grilling-me 收窄到软件规划工作区（这是有意裁剪，非缺陷，但未声明取舍）。
- **【低】缺会话预期管理**：未沿用原方法的「~45 分钟长会话」时长锚与「AI 版橡皮鸭调试」类比锚。
- **【低】横向定位独到但未显式声明**，用户难感知相对 rubber duck / Socratic / DDD 的优势。

### 视角四 · 兄弟 skill 边界与定位（P 维） — 73 / 良好

**总评**：自身边界声明相当扎实——「不做什么」清单明确点名 4 个兄弟作重定向目标，与 grilling-with-context 形成对称双向引用。扣分在三处缝隙：description 缺区分句、plan-create 衔接有缝、网络半孤岛。

**优点**
- **「不做什么」是边界即重定向表的范本级实践**：每条"不做"都附明确接管 skill（落盘→grilling-with-context、实现→plan-execute、规划→plan-create、搭脚手架→docs-workspace），可操作性强。
- **与 grilling-with-context 形成对称双向引用**，核心分叉锁定在「落盘与否」这一单一判据，是 7 个 skill 里最干净的一对关系。

**关键问题**
- **【高】description 缺少与 grilling-with-context 的区分句**：两 skill 共享核心触发词 'grill'/盘问。grilling-with-context 的 description 末尾专门加了区分句并回指 grilling-me，但 grilling-me 的 description 没有对等句——区分信息单向存在，触发词重叠时 skill 选择器在两者间歧义。**这是全报告唯一被标为"高"严重度的发现。**
- **【中】自定位 plan-create 上游，但 plan-create 把前置盘问硬绑到 grilling-with-context 且假设产物已落盘**：grilling-me 全程不落盘，其产物只是对话里的「共识摘要」，在 plan-create 流程里没有既定落点。
- **【中】在 skill 网络中是「半孤岛」**：除 grilling-with-context 外无兄弟回指；use-docs 路由表的盘问/术语行全部指向 grilling-with-context，用户经兄弟路径难发现 grilling-me。
- **【中】§3 把 plan-execute 与 plan-create 并列**，但无计划文件时直接跳 plan-execute 流程不通（plan-execute 硬前置是已存在的计划文件）。
- **【低】§0「不依赖 docs 系统」措辞有歧义**：可能被误解为连读都不读（本意是"不要求 docs 就位即可运行"，但 §2 用代码印证逻辑上应允许只读参考既有 ADR/spec）。
- **【低】完成标准「记录」在不落盘语境下易误解**为"写进文件"。

### 视角五 · 可执行性与 LLM 遵从度（A 维） — 70 / 良

**总评**：四条硬约束（一次一题/全程零写入/未达共识不动工/每题附推荐答案）锚定扎实、可观测、可自查，LLM 遵从度基础打得很好。但核心方法论整体下沉到条件加载文件，加载触发条件存在 Dunning-Kruger 缺口，完成标准又引用仅在 reference 中定义的「分支」，形成"不加载就无法自查完成"的循环。

**优点**
- **硬约束可观测、重复锚定、可自查**：四条硬约束在 description、§1、完成标准、不做什么多处重复且措辞一致，每条都有可观测外部表征（是否只问一题/是否动了文件/是否附推荐答案），便于每步自查与事后核验。
- 兄弟边界清晰、引用路径可用（两条 URI 均实测可解析）、渐进式披露结构干净。

**关键问题**
- **【中】条件加载触发器「不确定是否穷尽时先 read」自相矛盾，存在 Dunning-Kruger 缺口**：「不确定是否穷尽」是自指条件——尚未加载六类框架的 LLM 恰恰无法意识到自己可能漏问（不知道自己不知道）。结果是"最需要读该手册的会话，最不会触发加载"。
- **【中】完成标准的「每个分支」仅在条件加载的 reference 中定义，不加载则无法自查**：若 LLM 未加载 reference，完成标准引用的是上下文里没有定义的概念，自查变空转——LLM 会按即兴理解的「分支」宣布完成，绕过穷尽判据。与上一条叠加形成循环依赖。
- **【中】核心盘问节奏无端到端示例**：全文从未给出"用户陈述→提一个问题+推荐答案+理由→用户反馈→下一题"的完整对话示例，LLM 须从三处分散约束自行拼装格式，首次执行格式漂移概率高。
- **【低】§3 收敛「指出解决了哪个冲突/暴露了哪个新依赖」强制二选一**，对干净确认的分支易诱发捏造。
- **【低】「能查就别问」缺事实 vs 决策灰区裁决规则**（如"端点用 POST 还是 GET"既有 REST 约定又有团队偏好）。
- **【低】§0「建立现状心智模型（可选）」与 §1/§2 硬依赖有张力**：§1 能查就别问、§2 用代码印证都依赖已扫过代码，"可选"会让 LLM 跳过前置扫描。
- **【低】「唯一允许一次抛出的元问题」措辞有歧义**（可读作"元问题允许，一次一个"）。

## 三、交叉综合：多视角收敛的高置信度根因

> 以下每条都由 ≥2 个独立视角指向同一根因，置信度高于单一视角发现。这是本报告的增值分析，非简单拼凑。

### 根因 1（最高优先级）：渐进式披露「过度下沉」——入口丧失自包含性
**收敛视角**：S（§3/§4 孤儿未挂载）+ A（Dunning-Kruger 缺口 + 完成标准循环依赖）。

skill 把"本该是入口必备骨架"的内容当成了"可选深度"下沉到条件加载文件。后果链：
```
LLM 跳过加载（D-K 缺口）
  → 不知道六类（references §1）
  → 不知道推荐答案格式（references §3，且未挂载）
  → 不知道死锁/跑题逃逸（references §4，且未挂载）
  → 完成标准的「分支」无法自查 → 凭即兴宣布完成 → 穷尽判据失效
```
**修法**：①把六类清单**内联**进 SKILL.md §1 与完成标准（一行一类 + 典型问法）；②挂载 references §3/§4，或把死锁逃逸等关键失败路径上移入口；③加载条件去自指——改为"每次进入盘问前先 read"或直接内联。

### 根因 2：分类体系需在「入口 ↔ 下沉」间保持单一真源、且自身要 MECE
**收敛视角**：S（§2 入口 4 动作 vs references 5 探针未对齐）+ C（六类非严格 MECE）。

入口的施压摘要与 references 的探针细节是两套分类（4 vs 5，部分不对应），破坏了"入口须忠实预览细节"的披露契约；而六类自身又有 ②④、③⑥ 的重叠。两者同属"分类体系的一致性与完备性"问题。
**修法**：①入口 §2 改为 references §2 五探针的真实压缩摘要（或回填"用代码印证"为第六探针）；②六类明确"覆盖率清单（允许重叠）vs 穷尽判据（正交）"的二分，给重叠点归口规则。

### 根因 3：六类作为「通用穷尽判据」的完备性与领域适配
**收敛视角**：C（MECE + 缺权限 + 偏数据建模）+ A（完成标准需自包含，使六类缺陷直接威胁可靠性）。

一旦按根因 1 把六类内联进完成标准，六类的 MECE 缺陷就从"references 里的瑕疵"升级为"完成标准可靠性的直接威胁"。补权限/授权维度、声明领域裁剪指引变得必要。

### 根因 4：定位与衔接的「网络层」缝隙
**收敛视角**：P（独立，但与 SourceAlignment 的"使用时机丢失"共振）。

description 缺区分句 + 使用时机丢失，本质都是"skill 在网络中的可达性与路由精度"问题——用户在"想要被反驳""多决策依赖""定数据模型前"等典型痛点场景下，缺乏被自动路由到本 skill 的语义钩子。加上 plan-create 衔接缝隙与半孤岛，构成网络层的系统性短板。

### 根因 5：真源时效与归因精度
**收敛视角**：SourceAlignment（独立）。

源漂移（v1.1 重构、grill-me 被降级）+ 归因拔高（推荐答案=硬纪律）+ 使用时机丢失，影响"忠实度可复核性"与"路由精度"。原作者方法已演进，grilling-me 的归因停留在旧快照。

## 四、优先级改进清单

### P0（高置信度，多视角收敛，直接影响可执行性）
1. **入口自包含化**：六类清单内联进 SKILL.md §1 与完成标准；references §3/§4 补挂载指针或上移关键逃逸逻辑；加载条件去自指（默认加载或内联）。〔根因 1，S+A〕
2. **对齐施压分类**：SKILL.md §2 改为 references §2 五探针的真实压缩摘要，消除 4-vs-5 未对齐。〔根因 2，S〕
3. **description 补区分句**：补"只盘问不落盘；若要把术语/决策沉淀进 docs，用 grilling-with-context"。〔根因 4，P，唯一"高"严重度〕

### P1（重要，单视角强发现）
4. **六类收紧**：补「权限/授权/访问控制」为第七类或③子项；声明六类是"数据建模为中心的起点清单"+ 领域增补指引（演进/迁移/版本、一致性/时序、可观测性、多租户/合规/i18n 按需追加）；②④、③⑥ 给归口规则。〔根因 3，C〕
5. **补回 5 个具体使用时机**到 description/§0（写规格前/让 agent 实现前/定数据模型或 API 形状前/多决策相互依赖/想要被反驳而非附和）。〔根因 4+5，P+Source〕
6. **修正 §3 下游因果链**：plan-create 是新设计默认下游，plan-execute 仅在已有计划时适用，勿并列；补 plan-create 衔接说明（共识摘要可作为已澄清项起点，避免重复盘问）。〔根因 4，P〕
7. **更新源归因**：补版本锚点"对应 mattpocock skills v1.1（2026-07）抽出的 grilling 原语；原作者现更推荐 grill-with-docs/domain-model，本 skill 取其纯盘问子集"。〔根因 5，Source〕
8. **网络回指**：use-docs 路由表盘问/术语行补"纯压力测试不落盘 → grilling-me"分支。〔根因 4，P〕

### P2（打磨项，低严重度）
9. 加一段最简端到端示例（问题+推荐答案三段式+用户否决后继续走树）。〔A〕
10. §3 收敛「冲突/依赖」改为允许空（干净确认明说"无"）。〔A〕
11. 「能查就别问」补灰区裁决（默认先查，查不到或多解才交用户）；§0「可选」改为条件懒加载；元问题措辞去歧义；「代码」泛化为「既有上下文」。〔A+C〕
12. §0「不依赖 docs」收紧为"不写入、但可只读参考既有 docs"；完成标准「记录」改为"记录在共识摘要中（不落盘）"。〔P〕
13. 推荐答案「硬纪律」改述为"原方法便利做法，本 skill 强化成格式硬约束"；补 rubber-duck 类比/时长预期；显式横向定位声明；声明通用性收窄取舍。〔Source〕
