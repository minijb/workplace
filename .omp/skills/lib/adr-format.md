# ADR 格式（架构决策记录）

ADR（Architecture Decision Record）记录**难以回退、需要背景才看得懂、且经过真实权衡**的决策。存放于 `docs/adr/`，文件名按序编号：`0001-slug.md`、`0002-slug.md`。新增时同步登记到 `docs/adr/INDEX.md`。

## 模板

```md
# {决策简述}

{1-3 句话：背景是什么、决定了什么、为什么。}
```

ADR 可以只有一段。价值在于记录**「做过这个决定」与「为什么」**，而非填表。

## 可选段落

仅当确有价值时才加，多数 ADR 用不到：

- **Status**（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——决策会被复访时有用。
- **Considered Options**——被否决的方案值得被记住时。
- **Consequences**——存在非显而易见的下游影响时。

## 编号

扫描 `docs/adr/`，取现有最大编号 +1。新增后更新 `docs/adr/INDEX.md` 的一行说明。

## 何时创建

三者**同时**成立才写 ADR：

1. **难以回退**——日后改主意的成本可观。
2. **没有背景会困惑**——未来的读者会疑惑「为什么这么做」。
3. **真实权衡的结果**——存在合理的备选方案，且为特定理由选了其一。

任一不成立则跳过：容易回退的直接改；不令人困惑的无需记录；没有备选的只是「做了显然的事」。

## 什么算数

- **架构形态**：单体还是微服务；写模型是否事件溯源。
- **上下文间的集成方式**：通过领域事件还是同步 HTTP。
- **带来锁定的技术选型**：数据库、消息总线、认证提供方、部署目标——而非每个第三方库。
- **边界与范围**：客户数据归 Customer 上下文独占，其余上下文仅以 ID 引用。明确的「不做」同样有价值。
- **对显然路径的刻意偏离**：手写 SQL 而非 ORM，且理由充分。
- **代码中不可见的约束**：因合规不能用某云；因合同要求响应低于 200ms。
- **被否决的备选方案**：考虑过 GraphQL 却选 REST，理由微妙——记下，否则六个月后又会有人提议 GraphQL。
