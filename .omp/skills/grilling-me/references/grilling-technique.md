# 盘问技术手册（Grilling Technique）

> **适用范围**：本文件是 `grilling-me` 的按需加载参考，提供系统化盘问的具体手法。首次盘问、不确定是否已穷尽分支、或需要构造施压场景时先 `read` 本文件。本手册只讲「怎么盘问得透」，不涉及任何落盘动作（落盘归 `grilling-with-context`）。
>
> 访问方式：本 skill 目录内 `./references/grilling-technique.md`，或 `skill://grilling-me/references/grilling-technique.md`。

## 1. 分支覆盖（如何走遍设计树而不漏问）

盘问的穷尽判据是「设计树的每个分支都有结论」。为避免凭感觉漏问，按下列**六类**逐个确认；六类各有结论（或用户明确「跳过」）才算穷尽。这六类同时也是**问题清单的来源**——不知下一题问什么时，挑当前覆盖最薄的一类下手。

| # | 分支类别 | 要逼出的结论 | 典型问法 |
|---|---|---|---|
| ① | **概念定义** | 每个核心概念**是什么 / 不是什么**、边界在哪 | 「你说的 X，包含 Y 吗？排除 Z 吗？」 |
| ② | **关系模型** | 概念间如何**关联 / 组合 / 引用**，基数与归属 | 「一个 A 能有几个 B？B 归 A 独占还是可共享？」 |
| ③ | **生命周期与可变性** | 创建 / 流转 / 销毁路径，可变还是不可变 | 「X 谁来创建？能改吗？什么时候销毁？改了谁负责同步？」 |
| ④ | **存放与索引** | 落在哪里、如何被**检索**、唯一键是什么 | 「X 存哪？按什么查？同名/同键怎么去重？」 |
| ⑤ | **命名碰撞** | 同名异义 / 异名同义、与既有术语冲突 | 「这个 'session' 和登录 session 是一回事吗？」 |
| ⑥ | **边缘场景** | 空、极大、并发、失败回滚等**边界** | 「一条都没有时怎么办？百万条呢？两人同时改呢？中途失败回滚吗？」 |

**依赖排序**：先问被其它分支依赖最多的底层概念（通常是 ①②），再向上解依赖；否则上层决策会建在未定义的术语上。例如先定「Order 是什么」（①），才能问「Order 归谁」（②）。

**何时算穷尽**：六类各有结论，或用户对某类明确说「跳过」——后者要记下「该类未盘问」，不当作已解决。

## 2. 施压探针（怎么把模糊逼成精确）

每一题不止问，还要**主动施压**。下列探针按需取用，目标是把用户的模糊表述逼到「能在代码里落地」的精确度。

- **反例探针**：举一个看似符合定义、实则不该被纳入的例子，逼用户收紧边界。「那一个已退款但未关闭的 Order 算 'active' 吗？算的话你这定义就漏了。」
- **边界探针**：直接打 ⑥ 的边界——零值、极大值、并发、失败。「这个缓存永不过期吗？那脏数据怎么清？」
- **冲突探针**：把当下说法与既有陈述、既有代码、常理对撞。「你说所有支付都同步——但代码里这里是异步回调，哪个对？」
- **代价探针**：逼用户承认权衡的代价。「选不可变意味着每次改都重建——百万级数据下你接受这个成本吗？」
- **替换探针**：给一个更简单的替代方案，问为什么不选它，逼出真实约束。「为什么不用一张大宽表？是因为 schema 会爆吗？」

**力度拿捏**：探针是**逼精确**，不是抬杠。用户已说清楚的就不再纠缠；发现矛盾才升级为冲突探针。

## 3. 推荐答案怎么给

每个问题**都要**附推荐答案，这是 mattpocock 盘问法的硬纪律。给法的约束：

- **先结论、再一句理由、再点出代价/风险**：「我推荐 X，因为 Y；代价是 Z。」
- **依据优先用事实**：推荐答案尽量基于你查到的代码或既有结论；纯主观偏好要标注「[无事实依据，仅经验判断]」。
- **别越俎代庖**：推荐答案只是建议，**决策权在用户**。用户否决你的推荐，就接受其决策并继续走树，除非该决策与既有事实矛盾——矛盾时用冲突探针再确认一次。
- **可给多选**：当 A/B 两条路都合理且取决于用户偏好时，给两到三个选项 + 各自代价 + 你的推荐，但**最终仍只问一个问题**（「选哪条？」）。

## 4. 节奏与收口

- **一次一题是硬约束**：即使用户催「你一次问完」，也坚持逐题——一次多题会让推荐答案失去依据（后面的答案常依赖前面的决策）。可以预告「接下来还有 N 类要过」，但不并问。
- **用户疲劳**：盘问冗长时，每解决一个分支用一句话复述结论并标进度（「⑥边缘场景还剩并发与回滚两题」），让用户看到终点。
- **死锁（两决策互为前提）**：先凭推荐答案**临时假设**其中一个，继续走树，回头再验。明确告诉用户「我假设 A 成立继续，稍后回来验 A」。
- **琐碎分支**：明显无关紧要的分支（改了影响极小）不必深盘，推荐一个默认答案让用户快速确认即可，省下火力给关键决策。
- **跑题 / 范围蔓延**：用户扯到无关细节时，主动拉回：「这个先记下，我们先把 X 的定义定完。」不要让盘问无限发散。
- **收口**：六类覆盖完毕后，不复盘全部细节，只给「达成的关键决策 + 仍未解决的冲突 + 动工前待澄清事项」三段摘要，把「下一步」交还用户。
