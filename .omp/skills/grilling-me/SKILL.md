# 纯盘问（Grilling Me）

动工前，就一份计划、决策或想法对用户**逐题、无情地盘问**，沿设计的每个分支推进、逐个理清决策间的依赖，逼出隐藏假设与未决冲突，直到与用户**达成共识**。全程**只读、不落盘、不实现**——只问问题、给推荐答案、用代码印证。改编自 mattpocock 的 grilling：无情的逐题访谈 + 每题附推荐答案 + 事实自查 vs 决策交用户 + 共识达成前不动工。


## 0. 前置（轻量、只读）

本 skill **不依赖** docs 系统，也不创建任何文件。开始前只做两件**只读**的事：

1. **圈定盘问对象**：让用户点明要盘问的那份计划/决策/想法（一句话或一份草稿）。对象不清 → 先问「我们这次要压力测试的具体是什么？」，这是唯一允许一次抛出的元问题。
2. **建立现状心智模型（可选）**：`glob` / `grep` 盘问对象涉及的源码与既有上下文，以便盘问中用代码印证用户陈述（§2）。**查得到就去查，不要拿去问用户。**

## 1. 盘问节奏

- **一次一题**：每次只问**一个**问题，并附上你**推荐的答案**与一句理由，等用户反馈再继续。一次抛多题会让人无所适从。
- **遍历设计树、按依赖排序**：沿设计的每个分支逐题推进，先问**被依赖最多**的底层概念，再向上解依赖。**穷尽判据**：每个分支要么走到底有结论，要么用户明确说「这条先跳过」。如何系统化地走遍分支、避免漏问，见 `./references/grilling-technique.md` §1（六类分支覆盖）——**首次盘问或不确定是否穷尽时先 `read` 该文件**。
- **能查就别问**：一个问题若能通过探查代码库或工具回答（事实），就去探查，不要拿去问用户。**决策**才是用户的——逐个交给用户并等其回答。
- **未达成共识不动工**：在用户确认共识前，**不要开始实现**这份设计。本 skill 不落盘、不写代码、不改文件。

## 2. 盘问中主动施压

盘问不是只问，每一题都用下列动作逼出精确结论（探针库见 `./references/grilling-technique.md` §2）：

- **对撞既有结论**：用户当下的说法与其早先的陈述或常理冲突 → 当场揭出。「你前面说 X，但这里像是 Y——到底是哪个？」
- **收窄模糊用语**：用户用了模糊或过载的词 → 提出精确的规范说法。「你说 'account'——指 Customer 还是 User？这俩是不同的东西。」
- **拿具体场景施压**：构造探测边界的具体场景（尤其边缘情况：空、极大、并发、失败回滚），逼用户把边界说精确。「你说一个 Order 只属于一个 Customer——那合并下单、子账户代付时 Order 归谁？」
- **用代码印证**：用户陈述某事如何工作时，去查代码是否一致；发现矛盾就揭出。「你的代码取消的是整个 Order，但你刚说可以部分取消——哪个对？」

## 3. 收敛

- 每解决一个分支，用一句话复述结论，并指出它**解决了哪个冲突 / 暴露了哪个新依赖**。
- 全部分支走完、与用户确认共识后，给一份**共识摘要**：达成的关键决策、仍未解决的核心冲突、以及动工前还需澄清的事项（留给用户决定是否进入 `plan-create` 规划或 `plan-execute` 执行）。
- **任何未解决的核心冲突都要显式标注**，不得默默跳过。

## 完成标准

- 设计树的每个分支都被逐题盘问到，或有用户明确的「这条先跳过」。
- 每个问题都附了你推荐的答案与理由；事实性问题已自行查证、未转嫁给用户。
- 隐藏假设与未决冲突已被逼出并记录；仍未解决的核心冲突已显式标注，未默默跳过。
- 与用户就设计达成共识，或明确标注仍未解决的核心冲突与动工前待澄清事项。
- **全程零写入**：未创建、修改、删除任何文件（所有探查均为只读）。

## 不做什么

- **不落盘**——不创建/修改任何文件，尤其不写 docs 系统（`CONTEXT.md` / `docs/adr/` / `docs/spec/` 等）。要落盘用 `grilling-with-context`。
- **不实现设计、不写业务代码**——动工归 `plan-execute`；规划归 `plan-create`。
- **不搭建 docs 脚手架**——归 `docs-workspace`。
- **不替用户做决策**——决策交给用户；你只给推荐答案与依据。
- **不一次抛多个问题**，**不连续追问而忽略用户反馈**。
- 未与用户达成共识前，**不开始执行设计**。
