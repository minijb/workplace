# 模块级 CONTEXT.md 模板

> **适用**：单一限界上下文（一个模块/领域）。多上下文仓库中，每个上下文目录各用本模板建一份。
>
> **字段分两层**：**核心层**（必填，保持轻量）与**扩展层**（可选，整段可省略）。不确定要不要扩展字段时，就不写——核心层已足够。
>
> 访问方式：`./references/context-template-module.md` 或 `skill://context-system/references/context-template-project.md`。

---

```md
# {上下文名称}

{一两句话说明：这个上下文负责什么、为什么单独存在、它的边界在哪。}

## Scope

- **In**: {本上下文管辖的概念/能力，如「下单、订单状态机、取消」}
- **Out**: {显式声明不属于本上下文的内容，如「支付、库存、物流」}

## Language

### {TermName}

{一两句话定义「是什么」。}
- **Aliases**: {同一概念的其他叫法，选其中一个作主词}
- **Avoid**: {明确不要使用的近义词，避免歧义}
- **Example**: {最小实例帮助快速对齐，如「用户一次性购买 3 件商品生成的一个订单」}
- **See also**: {关联术语，如 Customer、LineItem}

<!-- ===== 扩展层（可选，以下整段可省略）===== -->
- **Status**: active
- **Source**: {出处，如 ADR-0001 / 2026-07 盘问 / spec-订单状态机}

<!-- 以下为可选字段，按需启用 -->
- **Owner**: {术语负责人}
- **Since**: {引入时间或版本}
- **Anti-example**: {常被误认为本概念但其实不是的例子，收紧边界}

## Relationships

{术语之间的关系速览，用于一眼看清模型骨架。可用基数标注。}

- 一个 **Customer** 拥有多个 **Order**（1..*）
- 一个 **Order** 包含多个 **LineItem**（1..*）
- **Order** 经取消流程进入 `Cancelled` 状态

## Open Questions

{尚未达成共识、待澄清的术语，每条标状态。}

- [unresolved] 「退款」是 Order 的状态，还是独立的聚合？
- [resolved] 「取消」是否区分用户取消与系统取消？→ 已定：区分
```

---

## 字段分层说明

| 层 | 字段 | 必填 | 作用 |
|---|---|---|---|
| **核心** | name / definition / Aliases / Avoid / Example / See also | 是 | 厘清概念边界、消歧、快速对齐——CONTEXT.md 的本职 |
| **扩展** | Status / Source | 否 | 管生命周期、可溯源（项目有 ADR/spec 时价值高） |
| **可选** | Owner / Since / Anti-example | 否 | 治理、演进追踪、收紧边界——仅多团队/复杂场景启用 |

**守纪律**：核心层永远是轻量的；扩展层按真实需要逐项启用，不要一次全开。每多一个字段都是维护成本。
